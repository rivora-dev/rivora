# RFC-000: Vision

**Status:** Draft (Foundational)  
**Target Version:** Foundation → v0.1  
**Authors:** Sergio Rovira

---

# Abstract

Rivora exists to help engineering teams understand their engineering system.

Modern engineering teams already have excellent tools for source control, CI/CD, cloud infrastructure, observability, incident management, and AI-assisted development. The challenge is no longer a lack of tooling—it is a lack of shared understanding across those tools.

**Rivora doesn't replace your engineering tools. It helps them work together as one engineering system.**

It continuously builds context across engineering activity, allowing teams to understand what changed, why it happened, and what they should do next using evidence rather than intuition.

Adaptive Reliability is the first flagship capability built on top of this foundation.

---

# Problem Statement

Today's engineering stack is fragmented.

Engineering teams already use GitHub, CI/CD, Kubernetes, cloud providers, observability platforms, AI coding agents, IDEs, and documentation systems.

Each tool excels within its own domain.

None of them share understanding.

GitHub does not know why production failed.

Observability does not know which pull request introduced the regression.

A coding agent does not know a similar incident happened three months ago.

Engineers become the integration layer between disconnected systems.

---

# Vision

Rivora creates shared engineering understanding.

Instead of replacing existing tools, Rivora observes engineering activity, builds durable engineering memory, derives knowledge from relationships, evaluates engineering outcomes, verifies proposed improvements, and learns over time.

The result is a continuously evolving understanding of the engineering system.

---

# Product Philosophy

## Engineering systems already exist.

Engineering teams should not replace the tools they already trust.

Rivora embraces the existing ecosystem.

## Shared understanding matters more than more tooling.

The value is not another dashboard.

The value is context.

## Evidence over intuition.

Recommendations should always be backed by observable engineering evidence.

## User-owned by default.

Engineering memory belongs to the user.

External systems remain systems of record.

## Humans remain responsible.

Rivora proposes.

Engineers decide.

# Positioning

## Core Message

> **Rivora doesn't replace your engineering tools. It helps them work together as one engineering system.**

## Supporting Message

---

# What Rivora Is

Rivora is a continuous engineering understanding layer that sits alongside the tools engineering teams already use.

It creates durable engineering memory, connects engineering context, and enables future capabilities to improve engineering workflows.

---

# What Rivora Is Not

Rivora is not:

- A replacement for GitHub
- A replacement for Kubernetes
- Another observability platform
- Another CI/CD system
- Another coding agent
- Another deployment platform

Those tools remain authoritative within their own domains.

---

# First Flagship Capability

Adaptive Reliability.

Adaptive Reliability demonstrates the architecture by continuously understanding infrastructure, deployments, incidents, alerts, and operational changes while helping teams investigate and improve reliability.

It is the first capability—not the final product.

---

# Long-Term Direction

The same architecture can later power:

- Adaptive Code Review
- Adaptive CI/CD
- Adaptive Security
- Adaptive Documentation
- Adaptive Developer Experience
- Adaptive Platform Engineering

Each capability shares the same foundation of memory, knowledge, evaluation, verification, improvement, and learning.

---

# Success

Rivora succeeds when engineering teams stop asking multiple disconnected tools independent questions and instead ask one engineering system for understanding.

**Help engineering teams understand their engineering system.**
